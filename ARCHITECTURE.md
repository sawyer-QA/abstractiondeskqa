# ARCHITECTURE.md — AbstractionDeskQA
> Living architecture document. Update with every change that alters structure, data flow, dependencies, or decisions. Each PR that changes architecture must touch this file, the changelog, and (if a decision was made) an ADR.

**Status date:** 2026-07-05 · **Baseline:** commit `4486061`

## 1. System Overview

Static multi-page toolkit for hospital core measure abstractors. GitHub Pages hosting, custom domain (CNAME), no build step, no server-side code owned by this repo. One page (lookup.html) consumes an external Google Apps Script API backed by Google Sheets.

```
┌────────────────────────────── abstractiondeskqa.com (GitHub Pages) ─┐
│ index  lookup  sep1  lkw  cmo  hbips  abstractly  404               │
│   │       │                                                        │
│   │       ├─ GET data/qa.json (same-origin static) ──► SEED fallback│
│   │       ├─ POST submissions ──► Google Apps Script ──► Sheets    │
│   │       └─ cache: localStorage (12h TTL)                         │
│   └─ feedback POST ──► formly.email (+ Cloudflare Turnstile)       │
│ all pages ──► GA4 · Google Fonts · phosphor icons (self-hosted) · BMC│
│                                                                      │
│ nightly (CI): scripts/sync-qa-data.mjs ─► Apps Script ─► Sheets     │
│   → PHI scrubber → data/qa.json (bot commit, scoped, main)         │
└─────────────────────────────────────────────────────────────────────┘
```

## 3. Data Flow — lookup.html

1. **Read (T-07):** Load → check `qc-sheets-cache-v1` (TTL 12h). Hit → render, refresh in background against `data/qa.json`. Miss → GET same-origin `data/qa.json` (static, regenerated nightly — see step 5).
2. Fetch failure or empty `entries` → `entries = [...SEED]` (embedded fallback), tag defaults merged, visible warning badge shown. There is intentionally no live-Sheets fallback tier between the static file and SEED — the whole point of T-07/TD-4 was to remove the read path's live-Sheets dependency, not add a rare-case one back in. **Known gap:** the background refresh's own fetch failures are still silent (no badge, no telemetry) — only the cold-load failure path is user-visible.
3. User submissions (add panel, paste modal) → client-side auto-tag detection → review chips → POST JSON to `SCRIPT_URL` → sheet row awaiting curator approval. **Known gap:** no bot protection or rate limit on this path (index form has Turnstile; this doesn't).
4. Render: full innerHTML template re-render; interactivity via inline onclick handlers calling globals.
5. **Nightly sync (T-07):** `.github/workflows/sync-qa-data.yml` (schedule + `workflow_dispatch`) runs `scripts/sync-qa-data.mjs`, which GETs the same two Apps Script endpoints (`SCRIPT_URL`, `SCRIPT_URL?type=tags`), runs a heuristic PHI scrubber over every entry, and writes `data/qa.json` (`{entries, tags, generatedAt}`). The workflow verifies the diff touches only `data/qa.json` before committing as `github-actions[bot]` directly to `main` — see ADR-0004 for the scoped exception to the normal PR-only rule.

## 5. Cross-Cutting Conventions (target state)

- **Output encoding:** single shared `esc()` escaping `& < > " '`, now living in `assets/site.js` (T-04 merged lookup.html's `esc()` and sep1-tool.html's `escH()` — the latter never escaped single quotes — closing F-18). Every page's dynamic-value escaping calls this one function.
- **Design tokens:** `assets/tokens.css` holds a light default `:root` (index/sep1-tool/hbips-tool/abstractly, plus lookup.html's structural/purple subset) and a `:root[data-theme="dark"]` override (lkw-tool/cmo-tool) (T-04). Two deliberate, documented carve-outs remain — not drift: **index.html's nav** stays page-local (a different design serving in-page scroll anchors, not cross-tool links — reskinning it would misrepresent its own navigation model); **lookup.html's typography and navy/teal/red/amber/green palette** stay page-local (only the canonical `--purple:#7B5CE5` accent and generic structural tokens converge — see the 2026-07-07 Why-This-Change note for the full reasoning).
- **Breakpoints:** 640px (mobile), 900px (tablet). (Baseline: 6 ad-hoc values.)
- **Events:** delegated `addEventListener`, no new inline `onclick`. T-04 advanced this in a bounded way: the tab-widget (`switchTab`/`showTab`) and share-strip copy-link are now delegated via `site.js`. T-05 converted lookup.html's ~55 non-button interactive elements (sidebar filters, difficulty/category chips, tag cloud/suggestions/review chips, card headers) to real `<button type="button">`s for keyboard operability, but deliberately kept their existing `onclick="fn()"` attributes rather than migrating to delegated listeners in the same pass — that migration, plus each tool page's page-specific onclick wiring (`gAns`, `toggleHelp`, quiz/case navigation, etc.), is now T-14's job.
- **Script load order:** `assets/site.js` is loaded synchronously (`<script src="assets/site.js"></script>`, no `defer`), placed immediately before each page's own inline `<script>` block — never a deferred `<head>` reference. Reason: some pages (e.g. sep1-tool.html's seed-data loop) call `esc()` synchronously at parse time, before `DOMContentLoaded`; a deferred site.js would not yet be loaded when that code runs, causing a `ReferenceError`. One uniform placement rule avoids a per-page judgment call each time a new page is added.
- **Spec logic:** pure functions in `assets/spec-logic/`, unit-tested against manual-cited cases; page files only wire UI.

## 7. ADR Index

- **[ADR-0001 — Single-file-per-tool, no build step](docs/adr/0001-single-file-per-tool-no-build-step.md).** *Accepted (retroactive).* Context: solo builder, GitHub Pages, tools must be trivially copyable/shareable. Consequence: duplication managed via shared `<link>`/`<script>` assets rather than a bundler; revisit if pages exceed ~10 or a framework need emerges.
- **[ADR-0002 — Google Sheets as content CMS](docs/adr/0002-google-sheets-as-content-cms.md).** *Accepted, amendment adopted (T-07).* Context: curator (non-dev) edits content; zero backend cost. Amendment: nightly export to versioned static JSON, Sheets remains authoring surface (fixes latency/quota, adds provenance/diffability) — implemented per ADR-0004.
- **[ADR-0003 — PHI-free by design](docs/adr/0003-phi-free-by-design.md).** *Accepted.* No user chart data is transmitted or stored; tools operate on user-entered abstractions client-side only. Any future feature that would transmit clinical text requires a new ADR.
- **[ADR-0004 — Static JSON pipeline with scoped bot-commit exception](docs/adr/0004-static-json-pipeline-and-scoped-bot-commit.md).** *Accepted.* lookup.html reads a nightly-regenerated `data/qa.json` instead of calling Apps Script live; the nightly workflow may commit directly to `main`, but only that one file, enforced by an automated scope guard, plus a PHI-pattern scrubber before every write.
- **[ADR-0005 — Accept 404.html's decorative watermark as a color-contrast exception](docs/adr/0005-404-watermark-contrast-exception.md).** *Accepted.* `.error-code`'s aria-hidden ghost-number watermark is a confirmed `color-contrast` false positive (redundant to real accessible error text); `lighthouserc.json` scopes the accessibility gate to `error` on the 7 tool pages with 404.html carved out at `warn`, rather than changing the page's markup or visuals.
- ADR template: `docs/adr/NNNN-title.md` → Context / Decision / Status / Consequences / Alternatives considered.

## 10. Why-This-Change Notes

(append-only; newest first)

- **2026-07-14** — T-20: Closed TD-18 (404.html's `.error-code` watermark color-contrast finding). A local Lighthouse run against the unmerged `engineering-backlog` working tree (`python -m http.server` + Chrome DevTools — not production, since PSI/the live domain would have graded the pre-backlog file) confirmed the page scores 95/100 with exactly one failing audit: `color-contrast` on the aria-hidden decorative watermark, already redundant to the real `<h1>`/`<p>` error text. No markup or visual change was made — `aria-hidden` doesn't exempt visual-contrast audits from evaluating rendered text (it addresses AT-tree exposure, not sighted low-vision legibility), so the finding is real but harmless. Resolved via **ADR-0005**: formalized as an accepted CI exception rather than converting the glyph to an image/SVG (T-20's original 2026-07-11 framing, not pursued — unnecessary complexity for a cosmetic-only false positive on the site's lowest-traffic page). `lighthouserc.json`'s `categories:accessibility` assertion moved from a flat global `warn` to `assertMatrix`: `error` at 0.95 for the 7 tool pages, `warn` at 0.95 for 404.html specifically.
- **2026-07-11** — T-18: Closed TD-17 (`role="button"` elements focusable but not keyboard-operable). Surfaced during T-17's `.how-box-hdr` fix but deliberately not folded into that ticket — a full sitewide inventory (grep for `role="button"`, `.help-btn`, `.case-card`) found the bug class in 4 files, not the 2 the original T-18 ticket text named: hbips-tool.html's 6 `.help-btn` spans had `role="button" tabindex="0"` but zero keydown handler; sep1-tool.html's and hbips-tool.html's `.case-card` divs had an Enter-only handler (`onkeypress`/`onkeydown`); lkw-tool.html's and cmo-tool.html's `.case-card` divs had no `role`/`tabindex`/keyboard handling at all — mouse-only, not even in the tab order. The same inventory also found `cmo-tool.html`'s 8 doc-type toggle divs and `sep1-tool.html`'s 6-element type-picker widget (5 options + 1 trigger) in the same "clickable div" shape — split out to **T-19** rather than fixed here, since neither is a plain button: the doc-type toggles are a multi-select (checkbox semantics) and the type-picker is a popover trigger+options pattern, both needing different ARIA roles than `role="button"`, not just a keydown handler.
  - **One shared delegated listener, not 10 inline `onkeydown` copies.** T-17's `.how-box-hdr` fix set an inline-`onkeydown` precedent, but repeating that string across 10 elements (6 identical help-btn instances, 4 case-cards) would have meant 10 near-duplicate copies and cut against §5's "delegated `addEventListener`, no new inline handlers" convention. Added one `document`-level `keydown` listener in `assets/site.js` that matches `e.target.closest('[role="button"]')` and synthesizes `.click()` on Enter or Space (`preventDefault()` on Space only, to stop page scroll) — the existing `onclick` handlers fire unchanged, so no page-specific JS needed editing beyond removing now-redundant partial handlers. The selector naturally excludes `role="tab"` elements (already handled by T-10/T-17's tablist keydown listeners) and the T-19 elements (neither carries `role="button"` yet) with no special-casing required. Also guards `e.repeat` (OS key-autorepeat would otherwise re-fire `.click()` repeatedly while a key is held) and `tagName === 'BUTTON'` (defensive — nothing in scope is a native button today, but a future redundant `role="button"` on a real `<button>` would otherwise double-fire the click).
  - **lkw-tool.html/cmo-tool.html's `.case-card` template literals gained `role="button" tabindex="0"`** (previously neither attribute existed) — this is the only markup change beyond removing redundant inline handlers; no CSS was needed since neither file has an `outline:none` rule touching `.case-card`, so the browser's default focus ring applies once the element is focusable.
  - **sep1-tool.html's `.case-card` template and hbips-tool.html's JS-built case-card (`renderCaseList()`) had their old Enter-only inline handler removed** (`onkeypress="if(event.key==='Enter')startCase(...)"` and `div.onkeydown = e => {...}` respectively) — leaving either in place alongside the new delegated listener would have double-fired the card's activation function on every Enter press.
  - Verified manually in-browser on all 4 pages (no automated a11y tooling run this session): Tab reaches every one of the 10 elements with a visible focus ring; Enter and Space both activate each one; Space does not scroll the page; holding either key down does not repeatedly re-trigger navigation; tab-strip Arrow/Home/End behavior (T-10/T-17) is unaffected; mouse click still works on all 10.
  - No ADR: this is a conformance fix to an already-decided pattern (keyboard-operable interactive elements per WCAG AA, the same category as T-10/T-12/T-17), not a new architectural decision — the delegated-listener-vs-inline-handler choice was a straightforward application of §5's existing convention, not a new alternative being weighed.

- **2026-07-09** — T-16: sitewide inline-style → component/utility class conversion, closing the ~230-instance gap T-14 split out at close-out. Full pattern-level audit done first (three parallel research passes, one per file group, each cross-referencing every inline `style=` against that file's own `<style>` block before proposing a class), then executed file-by-file: 404.html (2) → abstractly.html (3) → index.html (21 of 22, 1 left inside dead code) → hbips-tool.html (40) → sep1-tool.html (32) → cmo-tool.html (35 of 36) → lkw-tool.html (92 of 93). Zero static-markup `style=` attributes remain sitewide except 3 deliberate exceptions: index.html's commented-out newsletter block (not live markup — user confirmed leave-as-is), and one JS-driven quiz-progress-bar initial width each in lkw-tool.html/cmo-tool.html (`element.style.width` is overwritten at runtime by the quiz logic; the static value html-validate would flag is never what a user actually sees, so converting it buys nothing and risks the render code for no reason).
  - **Strategy, not a blanket utility-class framework:** every inline style got sorted into one of three buckets before touching markup. *Component-default* — when ~all instances of a class carried the identical override, the fix was changing the class's own default (e.g. hbips's `.section-label` margin-bottom was overridden to 12px at 9 of 9 call sites, so the class default moved from 10px→12px, deleting all 9 inline attributes with zero new CSS). *Utility-class* — a generic repeated value with no natural component home got one small, specifically-named class (e.g. `.gq-prompt`/`.gq-answers` for the guided-workflow question/answer pattern shared across sep1/lkw/cmo). *Semantic/one-off* — genuinely content-tied values (calculator field widths, lkw's 2-row priority-table clinical coloring) got a small scoped class named for what it *is*, not a generic utility. Deliberately did not build a `.mt-12`/`.fs-13`-style utility framework — CLAUDE.md's convention is named components/tokens, and a large generic utility set on a build-step-less site is its own form of sprawl.
  - **Two real design decisions, confirmed with the user before executing:** (1) the footer brand-letter "QA" span — `lkw-tool.html`/`cmo-tool.html` hardcoded the token's light-theme value (`#0da8a5`) even though both pages force `data-theme="dark"` (where the token resolves to `#14b8a6`); converged to `var(--teal2)` via one shared `.site-footer-brand span` rule in `assets/site.css` (fixes index/sep1/hbips/abstractly/lkw/cmo at once) plus a local equivalent in 404.html's own `<style>` (it doesn't link shared assets), closing the token drift as a byproduct — visible effect is lkw/cmo's footer letter shifting to match their own dark-theme accent, same category as T-04's flagged cmo purple-accent convergence. (2) 404.html's nav was rendering "QA" twice (an inline-styled span *and* the `.qa-badge` pill immediately after it) — leftover from an incomplete migration to the badge pattern; deleted the redundant span rather than just converting its color, matching index.html's clean single-badge nav.
  - **`.cx-icon` size+color, lkw-tool.html (16 instances) — preserved existing appearance rather than converging to cmo's convention.** Both files have identical `.cx-icon`/`.cx.variant` base rules, but lkw's icons carried a 100%-consistent inline override (18px, and a more saturated per-variant color than the box's own text color) that cmo's icons never had (cmo's icons just inherit the box's dimmer text color, no override). Converting lkw to match cmo would have been a real visual change to 16 elements nobody asked for; instead added explicit `.cx.warn .cx-icon{color:#fbbf24}`-style rules per variant, preserving lkw's current vivid-icon look exactly while still eliminating every inline instance. Noting the two files now have a deliberate, documented divergence in this one component rather than silently drifting.
  - **`display:none` → CSS class default, 3 elements (sep1-tool.html `#quizDone`/`#caseDetail`, hbips-tool.html `#quizScenario`).** Verified first that the JS toggling each element always sets `element.style.display` directly (never reads/toggles a class) — confirmed via grep on all three before touching markup — so moving the *default* hidden state from an inline attribute into a CSS class rule is a pure no-op for behavior; inline JS-applied styles always win over CSS class rules regardless. `#caseDetail` in sep1 gained a new `class="case-detail"` matching the identical convention hbips-tool.html already used for its own case-detail toggle.
  - **`.qr-v--bold` modifier, not a base-class change (cmo-tool.html's exclusion matrix, 10 cells).** The existing `.qr-v.y/.n/.w` 3-state class trio (already used 17× elsewhere in the file) was the obvious fit for the matrix's Yes/No/Review coloring, but the matrix cells also needed `font-weight:700` that the 17 existing plain usages don't carry — added as a separate `.qr-v--bold` modifier applied only to the matrix, rather than bolding the base class and silently changing the look of the other 17.
